Team Task Manager - Demo Video Teleprompter Script

Suggested video length: 5 to 7 minutes

Before recording, keep these ready:
- Live application opened in the browser
- GitHub repository opened in another tab
- Code editor opened with the project folder
- One admin account ready for login
- One normal user or employee account ready for showing assigned work


1. Start with the project introduction

Show on screen:
Open the live application home/login page.

Say:
Hello, my name is Atul, and this is my project Team Task Manager. It is a full-stack MERN application made for managing team projects, tasks, attendance, profiles, and notifications from one place.

The reason I built this project is that in a real team, work is not only about creating tasks. We also need user approval, role-based access, attendance tracking, task status updates, and a way for admins to communicate with team members. So I tried to bring all those features into one practical web application.


2. Explain the tech stack

Show on screen:
Open the GitHub repository, then show the frontend and backend folders.

Say:
Here I have divided the project into two main parts. The frontend is built with React and Vite, and the backend is built with Node.js and Express. MongoDB is used as the database, and Mongoose is used for creating models and relationships between users, projects, tasks, attendance, and notifications.

For authentication, I used JWT tokens. Passwords are secured using bcrypt hashing. I also used validation middleware on the backend so that wrong or incomplete data does not directly go into the database.


3. Show signup and login

Show on screen:
Go to the signup page.

Say:
Now I will show the authentication flow. On the signup page, a person can create an account by entering their name, email, mobile number, password, and role.

If the user selects Employee, then the employee ID field is also required. This is useful because an employee can later log in using email, mobile number, or employee ID.

Show on screen:
Select different roles in the signup form.

Say:
The app supports Admin, User, and Employee roles. After signup, the account does not directly get access. It goes into pending approval, and only an admin can approve it. This makes the flow more realistic because in an actual company system, new users should not get access without approval.

Show on screen:
Go to login page.

Say:
On the login page, the user first selects the role and then enters email, mobile number, or employee ID with password. After successful login, the app stores the JWT token and opens the protected dashboard.


4. Show admin dashboard

Show on screen:
Log in as admin and open the dashboard.

Say:
This is the admin dashboard. At the top, I am showing live time and date with a greeting. Below that, there are punch in and punch out buttons, task summary cards, my assigned tasks, and attendance information.

The summary cards show total assigned tasks, pending tasks, completed tasks, and overdue tasks. This gives the user a quick idea of what needs attention today.

Show on screen:
Point to punch in and punch out.

Say:
Here I have added attendance tracking. A user can punch in when starting work and punch out when work is finished. For admin, the dashboard can also show attendance details for team members, including present days and today's punch timing.


5. Explain admin login alerts

Show on screen:
If visible, show the admin login alert section. If not visible, explain from the dashboard area.

Say:
I also added admin login alert handling. If there are wrong password attempts for an admin account, the backend records those attempts. After multiple wrong attempts, the account can be blocked until password reset. This was added to show that I considered basic account security, especially for admin access.


6. Show sidebar and layout

Show on screen:
Point to the sidebar.

Say:
The main layout has a sidebar with navigation. From here, users can go to Profile, Dashboard, and Projects. If the logged-in user is admin, two extra sections are visible: Manage Employee and Manage Users.

There is also a small weather card in the sidebar. It uses the browser location if permission is available, otherwise it shows weather using a default location.


7. Show notifications panel

Show on screen:
Point to the right-side notification panel.

Say:
On the right side, I have a notification panel. Admin can send a notification by entering a title and message. Users can see those notifications and mark them as read.

This panel also shows pending account approvals for admin. So when a new user signs up, the admin can approve that account from here.

Show on screen:
Point to welcome new member section.

Say:
There is also a welcome section for new members. When someone punches in for the first time, they can appear here as a new member. This makes the dashboard feel more like a team workspace.


8. Show profile page

Show on screen:
Open Profile page.

Say:
This is the profile page. It shows the user's name, role or designation, profile picture, productivity percentage, and how long the user has been serving in the company.

For admin users, profile picture upload is available. The profile details are stored in MongoDB and loaded into the frontend through protected API calls.


9. Show manage employees and manage users

Show on screen:
Open Manage Employee page, then Manage Users page.

Say:
These sections are available only for admin. Here, the admin can select an employee or user and update profile details like name, designation, profile picture, and productivity.

I separated employees and users because in a real organization, different types of accounts may need to be managed separately. The productivity slider and profile preview make this page more interactive and useful.


10. Show projects page

Show on screen:
Open Projects page.

Say:
Now I am opening the Projects page. Admin can create a project by entering project name, description, and selecting which role the project is created for.

Show on screen:
Use the role dropdown and member checkboxes.

Say:
When I select the target role, the member list changes according to that role. This helps prevent assigning a project to the wrong group of people.

Show on screen:
Create or point to an existing project card.

Say:
Each project card shows the project name, description, target role, number of members, and an open button. Admin can also delete a project if needed.


11. Show project details and tasks

Show on screen:
Open one project.

Say:
Inside a project, the admin can update project details and manage the team members. Below that, the admin can create tasks.

Show on screen:
Point to create task form.

Say:
A task has a title, due date, status, description, and assigned users. A task can be assigned to one or more people from the project team.

Show on screen:
Point to filters and task cards.

Say:
Tasks can be filtered by status or assignee. The supported statuses are Todo, In Progress, and Done. Users can update the status of their assigned tasks, while admin can also delete tasks.

This section shows the main purpose of the project: creating work, assigning it clearly, and tracking progress.


12. Show forgot password

Show on screen:
Go back to login and open Forgot password.

Say:
I also added a forgot password flow. The user enters the registered mobile number, receives or gets an OTP depending on SMS configuration, verifies the OTP, and then creates a new password.

On the backend, the OTP is hashed before saving, it has an expiry time, and there is a limit on wrong attempts. The new password also cannot be the same as the old password.


13. Show backend code structure

Show on screen:
Open the backend/src folder in the code editor.

Say:
Now I will quickly show the backend structure. The backend is organized into models, controllers, routes, middleware, validators, and utilities.

Show on screen:
Open models folder.

Say:
The models define the database structure for users, projects, tasks, attendance, notifications, and admin login alerts.

Show on screen:
Open routes and controllers.

Say:
Routes define the API endpoints, and controllers contain the main logic for each feature. Middleware is used for authentication, role authorization, validation, database checks, and error handling.

This structure keeps the backend easier to maintain because every feature has its own place.


14. Show deployment setup

Show on screen:
Open package.json and railway.json.

Say:
The project is also ready for deployment. In the root package.json, I have scripts for installing dependencies, running locally, building the project, and starting it in production.

For production, the React app is built into frontend/dist, and the Express server serves both the API and the frontend. The railway.json file keeps the Railway deployment simple with build and start commands.


15. Closing

Show on screen:
Return to the live dashboard.

Say:
So this is my Team Task Manager project. It includes login and signup, admin approval, role-based access, project management, task assignment, attendance tracking, profile management, notifications, forgot password with OTP, backend API structure, and deployment support.

The main thing I focused on was making the project feel like a real internal team tool, not just a basic task CRUD app. Admins can control the workflow, and users or employees can focus on their assigned work.

Thank you for watching my demo.
